# Plan: govbot

> Source PRD: plans/fleet-monitor-prd.md (fleet-monitor observability, #36)

This is the project's master plan: a durable architectural header plus an ordered list of task
pointers. Each task is one feature on its own branch, ending in a PR. Task bodies live in
`plans/tasks/`; finished tasks move to `plans/tasks/done/`.

## Workflow

- New work is added by the `to-plan` skill: a self-contained task file under
  `plans/tasks/NNNN-<slug>.md` plus a pointer below. It appends; it never creates a second plan.
- `implement-next-task` takes the first eligible pointer (or an explicit task argument), builds it
  on its branch — AFK via `tdd`, `[decision]` via `talk-it-through`, `[verify]` paused for manual
  confirmation — runs `task-review`, then opens the PR after approval and flips the pointer to `[>]`.
- A pointer has four states: `[ ]` todo · `[~]` in progress (claimed) · `[>]` done, PR open,
  awaiting merge · `[x]` merged to `main`. `sync-main` flips `[>]→[x]` and moves the task file to
  `tasks/done/` once the PR merges.
- Pointers carry their direct prerequisites as an `(after NNNN, …)` suffix (none = no suffix). A
  task is selectable only once every ordinal in its `(after …)` list is **`[x]` (merged)** — so a
  dependent never branches off `main` before its prerequisite is actually on `main`.
  `implement-next-task --worktree` uses this to pick the first task not blocked by in-progress work,
  or reports "no independent task" when every remaining task is blocked.

## Architectural decisions

Durable decisions that apply across all tasks:

- **Monorepo shape**: self-contained modules under `actions/`, each runnable as a plain CLI script
  and as a GitHub Action (`action.yml`). Python actions use a `click` CLI entry point (`main.py`)
  with pipenv for dependencies; snapshot tests live in a snapshots folder regenerated by a
  `render-snapshots` script and are the source of truth for expected behavior.
- **Fleet config**: the pipeline-manager YAML files (`actions/pipeline-manager/chn-openstates-scrape.yml`
  and `chn-openstates-files.yml`, validated by `config.schema.json`) are the authority on the fleet —
  locales, orgs, repos, expected workflows. Paused/active is derived from each locale's `template`
  value (`*-paused` suffix = out-of-session); `disabled_jobs` switches off expected workflows per locale.
- **Fleet monitor**: new module `actions/fleet-monitor` (Python), decomposed per the PRD into
  config reader, poller, log harvester, shippers, watermark store, orchestrator, and Grafana assets.
  Runs as an hourly workflow (`cron: "0 * * * *"`) in this monorepo on GitHub-hosted runners.
- **Transport**: direct HTTP from Python to Grafana Cloud push endpoints (metrics + Loki). No
  Vector, no long-running collector.
- **Auth**: workflow default `github.token` for all GitHub reads (public data only). One repository
  secret: the Grafana Cloud write key.
- **Cardinality**: metric series carry state/org/workflow/paused labels only; Loki labels limited to
  org, state, workflow, outcome — run/job ids travel in the log line, not labels. Budgets stay
  inside Grafana Cloud free tier (≈10k series, 50 GB/mo, 14-day retention — re-verify at signup).
- **Alerting**: three launch rules — active-state run failed; active-state data-commit age > 48 h;
  collector heartbeat absent 3 h. Contact points: Slack webhook + email. Paused states never alert.
- **Testing**: snapshot tests for config reader, log harvester, and shipper payloads. The poller is
  deliberately untested at launch (pass-through against a live API).
- **Rollout**: build and validate in a fork against the real orgs' public data with a personal
  Grafana Cloud account; the ask to maintainers is one PR plus one secret.
